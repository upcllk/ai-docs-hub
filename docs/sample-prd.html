AI 文档协作平台

作者：产品团队 创建：2026-06-28 状态：评审中 文档 ID：sample-prd

一、背景与目标

随着 AI 工具的普及，团队越来越多地使用 AI 直接生成产品需求文档（PRD）和技术文档。这些文档通常以静态 HTML 或 Markdown 格式输出并保存在本地，导致以下问题：

  • 文档散落在各人电脑，无法统一访问
  • 版本更新后难以追踪变化，接收方需要重新分发
  • 缺乏类似飞书的行内评论机制，反馈流程割裂

本项目旨在构建一套轻量级的文档托管与协作系统，将 AI 生成的文档发布到 GitHub Pages，并实现基于文字选中的行内标注功能。

ℹ️
本期目标为完成 MVP Demo，核心验证"版本管理 + 行内标注"两个核心链路，不涉及用户权限管理和私有仓库访问控制。

二、用户与使用场景

角色主要诉求典型场景
产品经理 发布和管理 PRD 版本，收集技术团队反馈 将 AI 生成的 PRD 推送到仓库并打 Tag 发布
研发工程师 查阅 PRD，针对具体文字提出疑问或确认 选中某段需求描述，添加"这里的边界条件需要明确"
测试工程师 核对验收标准，标注测试关注点 选中验收标准章节，标注"需要补充异常路径测试用例"

三、功能需求

3.1 文档发布与版本管理

产品经理将 AI 生成的 HTML 文档放入仓库的 docs/ 目录，通过 Git Tag 标记版本后推送，GitHub Actions 自动完成构建与部署。

  • 每个 Tag（如 v1.0）对应一个独立的在线访问地址
  • 最新版本同时部署到根路径，供快速访问
  • 旧版本永久保留，不受新版本发布影响
⚠️
待确认：多文档场景下，文档之间是否需要索引页？当前设计每个文档独立，通过 URL 直接访问。

3.2 行内文字标注

在发布后的文档页面上，评审人可以选中任意文字片段并添加评论。标注以高亮形式展示在原文中，评论内容显示在右侧面板。

功能点优先级说明
选中文字弹出评论入口P0选中后显示浮动「添加评论」按钮
评论输入与提交P0弹窗输入，提交后写入本地 JSON 并提示 git 操作
高亮渲染P0页面加载时自动在原文标注已有评论的位置
侧边评论面板P0展示所有标注，点击可定位到对应高亮
回复功能P1对某条评论追加回复，Demo 阶段支持直接编辑 JSON
删除标注P2作者可删除自己的标注

3.3 版本间标注隔离

不同版本的文档拥有独立的标注集合，互不干扰。v1.0 的评论不会出现在 v1.1 的页面上，实现"版本快照 + 评论快照"的完整存档。

四、非功能需求

  • 无后端：全部基于静态文件托管，零服务器运维成本
  • 无强制鉴权：Demo 阶段读取完全公开，写入由本地 git 操作保证
  • 可替换性：标注后端通过 AnnotationProvider 接口抽象，后期可切换至 GitHub Issues 或自建服务
  • 兼容性：支持 Chrome 90+、Firefox 90+、Safari 14+

五、验收标准

  1. 执行 git tag v1.0 && git push origin v1.0 后，5 分钟内文档在 GitHub Pages 可访问
  2. 在文档页面选中文字，能弹出评论输入框，提交后页面出现对应高亮
  3. 刷新页面后，历史标注的高亮和侧边面板评论均正常展示
  4. v1.0 的标注不出现在 v1.1 的页面上
